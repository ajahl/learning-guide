Your assignment

A transit authority is weeks from approving an autonomous shuttle for a downtown route. Before it carries a single passenger, someone has to answer one question: when a pedestrian steps off the curb at dusk, will the shuttle's sensors actually see them?

You don't test that on a live street. You test it in GEMSTAR — and you've just joined the team that runs it. The trouble: GEMSTAR is a machine of seven interlocking Unreal plugins, and today it's a black box to you. Over the next ~40 focused hours your job is to understand it well enough to build that pedestrian-detection scenario yourself — and to add the one sensor parameter the authority keeps asking about.

Each phase below is a chapter on that path. By the last one you won't be reading the machine — you'll be changing it.

How to use: Work phases top→bottom. Tick each task; the bar fills as you go. Quizzes give instant feedback; checkpoints hide the answer until you click. Open file links jump to the real source in VS Code — they auto-detect your repo location, so they work on Windows or macOS as long as you open this guide from inside the cloned repo (learning-guide/). Platform differences (build toolchain, what runs) are summarized in the table at the bottom — works on both; Windows runs the most.

Which statement best describes GEMSTAR's architecture?
Checkpoint — reveal model answer Simulators (esmini/SUMO) produce vehicle & traffic-light data through dispatchers; consumers (vehicle manager, logger, visualizer) subscribe via update interfaces. Sensors (camera/lidar) produce sensor data that the logger writes as OSI traces and the visualizer renders. AGEMSTARRoot reads DefaultGemstar.ini and wires it all together. Plugins couple only to gemstar_interfaces.

Why does GEMSTAR lean so heavily on UInterfaces?
Checkpoint — reveal A UE interface (UINTERFACE + I... class) integrates with the reflection/Blueprint system and can be implemented by UObjects and queried at runtime — unlike a plain C++ abstract class. GEMSTAR uses them so a producer (e.g. ASUMOManager) only knows the IVehicleDataDispatcher contract, never the consumer's concrete type.

Linking Native Third-Party Libraries

GEMSTAR's signature complexity — and why it builds on Windows but not (yet) on macOS/Linux · ~3–5 h
▾
Chapter 2½ · The Foreign EnginesTwo of the machine's most powerful parts aren't Unreal at all — esmini and SUMO are foreign engines bolted on through Build.cs. On Windows they're shipped and ready; on macOS/Linux they're the reason the machine won't fully start. Meet them here.

This is the part most UE tutorials never cover, yet it's central to GEMSTAR: linking precompiled native libraries (esmini, SUMO, OSI, protobuf) into UE modules via Build.cs. On Windows the libs ship in the repo and everything links; on macOS/Linux this is exactly where the build breaks — so understanding it is the prerequisite to any port. Either way it's the same mechanism worth knowing.

The four moving parts in a ThirdParty Build.cs

CallWhat it does
PublicIncludePaths.Add(...)Lets your C++ #include the library's headers (e.g. esminiLib.hpp, osi3/*.pb.h)
PublicAdditionalLibraries.Add(...)Links the static lib (.lib/.a/.dylib) at build time — the actual symbols
RuntimeDependencies.Add(...)Copies a runtime DLL (e.g. esminiLib.dll) next to the binary so it loads at run time
PublicDefinitions.Add(...)Compile flags the lib needs — e.g. GOOGLE_PROTOBUF_NO_RTTI (protobuf clashes with UE's RTTI settings)
Why it builds on Windows but not macOS — see it for yourself. Open the Build.cs files below. Every third-party link sits inside if (Target.Platform == UnrealTargetPlatform.Win64) — so on Windows the libs link and the project runs. The Mac branch is empty/commented and there are no .dylib/.a Mac binaries in the repo, so on macOS the headers include but nothing links → unresolved symbols. Not a five-minute bug; a porting task.

Important — don't be misled by the standard tutorials. The idiomatic Unreal way to initialize a Game Feature is a UGameFeatureAction attached to a GameFeatureData asset (that's what the videos below teach). GEMSTAR does NOT use that. Instead each feature has a GEMSTAR_*_Loader that is a plain UActorComponent on BP_GEMSTAR_Root; on BeginPlay it casts the owner to AGEMSTARRoot, checks a Use* bool, and SpawnActors the manager. Learn the standard way for context, but the walkthrough teaches the pattern GEMSTAR actually uses. (The “Standard vs GEMSTAR” subsection in the walkthrough lays them side by side.)

PlatformBuilds & runs?Toolchain / notes
Windows (the documented platform)✅ Everything — incl. esmini & SUMOVS 2022 · PostGitPull_ReGenVSFiles.bat → GEMSTAR.sln. The Win64 libs ship in the repo. This is the smoothest path.
macOS⚠️ Pure-UE features only (lidar, visualizer, vehicle, camera)Xcode · PostGitPull_ReGenXcodeFiles.command → .xcworkspace. esmini/SUMO/OSI/protobuf ship Windows-only binaries (no .dylib) — those features need a port (Phase 2.5).

On Windows, ignore the "do it as a paper trace" notes — you can build and run for real. On macOS (e.g. an Intel Mac with a UE source build), stick to the lidar/visualizer path or take on the Phase 2.5 port.
Epilogue · The answer

Six chapters ago GEMSTAR was a black box and the authority's question was someone else's problem. Now you can place BP_GEMSTAR_Root in a map, point a lidar at a curb in a config file, trace the ray that hits the pedestrian, read the OSI message it produces — and change the one parameter that decides whether the shuttle sees them in time.

That's the whole job: turning "will it see the pedestrian?" from a guess into something you can run, read, and adjust. The machine isn't a black box anymore — it's a tool in your hands.
