Exercise 1 — A loader component that spawns a manager ★ the real GEMSTAR pattern

Intermediate~50 min Skills: UActorComponent, GetOwner, Cast<>, SpawnActor, config-gated init

Objective: Reproduce GEMSTAR_Lidar_Loader in miniature: a component on a "root" actor that, on BeginPlay, checks a bool and spawns another actor.

Steps

// MyRoot.h
UCLASS()
class CPPSCRATCH_API AMyRoot : public AActor
{
    GENERATED_BODY()
public:
    UPROPERTY(EditAnywhere, Category = "Toy")
    bool bUseManager = true;        // mirrors GEMSTARRoot::UseLidar
};
// MyLoader.h
UCLASS(ClassGroup=(Custom), meta=(BlueprintSpawnableComponent))
class CPPSCRATCH_API UMyLoader : public UActorComponent
{
    GENERATED_BODY()
protected:
    virtual void BeginPlay() override;
};
// MyLoader.cpp
#include "MyLoader.h"
#include "MyRoot.h"
#include "MyManager.h"

void UMyLoader::BeginPlay()
{
    Super::BeginPlay();

    // 1. Who owns me? (the component is attached to AMyRoot)
    AMyRoot* Root = Cast<AMyRoot>(GetOwner());
    if (!Root)
    {
        UE_LOG(LogTemp, Error, TEXT("Loader not on AMyRoot"));
        return;
    }
    // 2. Config gate — the Use* bool
    if (Root->bUseManager)
    {
        // 3. Spawn the manager into the world
        FActorSpawnParameters Params;
        AMyManager* Mgr = GetWorld()->SpawnActor<AMyManager>(
            AMyManager::StaticClass(), FTransform::Identity, Params);
        UE_LOG(LogTemp, Warning, TEXT("Loader spawned manager: %s"),
               *GetNameSafe(Mgr));
        // 4. (GEMSTAR then calls Root->RegisterManagerAndGetConfiguration(Mgr)
        //     and Mgr->Configure(...) — the config hand-off.)
    }
}
With the flag on: "Loader spawned manager…" logs, then the manager's own BeginPlay logs. With it off: neither. You've reproduced GEMSTAR's exact init mechanism — component → Cast owner → check Use* → SpawnActor.
Troubleshooting
GetOwner() returns null / cast failsThe component must be attached to an AMyRoot instance. In GEMSTAR the loaders live on BP_GEMSTAR_Root.
Component not in "Add Component" listNeeds meta=(BlueprintSpawnableComponent) and a successful build.
SpawnActor returns nullNeeds a valid UWorld (only during play) and a non-abstract class.
Open GEMSTAR_Lidar_Loader.cpp and match each numbered comment to a real line. The only thing you skipped is the config hand-off (RegisterManagerAndGetConfiguration + Configure) — find those two calls.

Standard UE vs GEMSTAR — the same job, two mechanisms

Both achieve "when this feature is on, set it up." Knowing both means you won't be confused by tutorials or by this codebase.

Standard UE (tutorials, Epic samples)GEMSTAR (this repo)
Init hookUGameFeatureAction on a GameFeatureData assetGEMSTAR_*_Loader, a plain UActorComponent
TriggerFeature state → Active (engine drives it)Owner actor's BeginPlay
Lives whereThe plugin's GameFeatureData assetA component on BP_GEMSTAR_Root in the level
On / off switchActivate/deactivate the featureA Use* bool on AGEMSTARRoot (from config)
Does the workOnGameFeatureActivating(...)Cast<AGEMSTARRoot>(GetOwner()) → SpawnActor → Configure
Why did GEMSTAR choose the component pattern? Likely simplicity and explicitness: putting one BP_GEMSTAR_Root in a map with loader components is easy to reason about and lets a level/Blueprint author add features without touching GameFeatureData assets. The trade-off: it leans on actor placement rather than the engine's feature-state system. Both are valid; just know which you're reading.
If a teammate says "add a GameFeatureAction to initialize the lidar," what do you correct them on? (GEMSTAR initializes via the loader component on the Root + a Use* bool — not GameFeatureActions.)

Exercise 2 — Read your own value from a config .ini

Intermediate~30 min Skills: GConfig, .ini sections, config-driven behavior

Objective: Read a custom key from an .ini at runtime — the mechanism AGEMSTARRoot uses to decide which features to enable and how sensors are configured.

Steps

[/Script/MyToy.Settings]
GreetingCount=3
GreetingText=Hello from config
int32 Count = 0;
FString Text;
GConfig->GetInt(TEXT("/Script/MyToy.Settings"),  TEXT("GreetingCount"), Count, GGameIni);
GConfig->GetString(TEXT("/Script/MyToy.Settings"), TEXT("GreetingText"),  Text,  GGameIni);

for (int32 i = 0; i < Count; ++i)
{
    UE_LOG(LogTemp, Warning, TEXT("%d: %s"), i, *Text);
}
The greeting logs exactly GreetingCount times. Editing the ini changes behavior without touching code — the whole point of config-driven design.
Open DefaultGemstar.ini and GemstarLidarTest.ini. Identify the section/key pattern. You now understand how GEMSTAR turns a sensor's FOV/range/ray-count into runtime behavior — and you'll edit exactly these in Phase 6.
