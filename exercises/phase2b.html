Phase 2.5 — Linking Native Third-Party Libraries: Worksheet

How GEMSTAR pulls esmini, SUMO, OSI & protobuf into Unreal via Build.cs — why it builds on Windows and what a macOS/Linux port would take. Mostly reading; doable on any OS.

↪ The assignmentThe obstacle from the story, up close: the foreign engines (esmini/SUMO) are shipped and ready on Windows — and are exactly why the machine won't fully start on macOS/Linux. Learn the wiring and the wall stops being a mystery.
Why this phase exists: the architecture (Phases 0–4) is the "what." This is the "how it physically builds." Sensor-sim projects live or die on native-library integration, and it's the single thing standing between you and a Mac build — so it deserves its own pass.

Exercise 1 — Dissect a ThirdParty Build.cs (esmini)

Intermediate~40 min Skills: PublicIncludePaths, PublicAdditionalLibraries, RuntimeDependencies, platform branches

Objective: Identify the four jobs a Build.cs does to wire in a precompiled library.

Steps

You can point at one line for each of: include path, static-lib link, runtime DLL copy. And you can see the Win64-only gate with your own eyes.
Why three different mechanisms?
Include pathCompile-time: so headers resolve. No code linked yet.
AdditionalLibrariesLink-time: resolves the symbols your code calls (static .lib/.a).
RuntimeDependenciesRun-time: ships the .dll so the OS can load it when the game runs. A static link alone isn't enough when the lib is a DLL.
If you linked the .lib but forgot RuntimeDependencies, when would it fail — build or launch? (Launch: "DLL not found." The build succeeds.)

Exercise 2 — OSI + protobuf in the interfaces module

Intermediate~30 min Skills: static libs, compile definitions, the RTTI gotcha

Objective: See the same pattern for the libraries at the heart of GEMSTAR — and one UE-specific gotcha.

Steps

The GOOGLE_PROTOBUF_NO_RTTI gotcha Unreal typically builds with C++ RTTI disabled. Protobuf can use RTTI by default; if its headers and UE disagree about RTTI you get ODR/link errors or crashes. Defining GOOGLE_PROTOBUF_NO_RTTI tells protobuf to match UE. This is a classic "integrating a mature C++ lib into UE" adjustment — exactly the kind of thing a Mac port must also get right.
Because these links are Public, who else can use OSI types? (Every module that depends on GEMSTAR_Interfaces — i.e. all the features. That's why the OSI headers "just work" in the loggers/managers.)

Exercise 3 — Scope the macOS port

Advanced~30 min (design) Skills: cross-platform porting, honest scoping

Objective: Turn "it builds on Windows but not macOS/Linux" into a concrete, sized task list. (You don't have to do the port — just scope it accurately. Skip this exercise if you're on Windows and just want it running.)

Checklist — what a macOS build actually needs

Honest verdict: this is a real porting project (days, not minutes), gated mostly on obtaining macOS builds of esmini/SUMO/OSI. If you just want GEMSTAR running, Windows needs none of this. On Mac, the no-port learning path is the lidar/visualizer features, which use only UE built-ins.
In one sentence to a colleague: why does GEMSTAR build on Windows but not macOS, and what's the smallest first step toward a Mac build? (Third-party links are Win64-only with no Mac binaries shipped; first step is obtaining/​building the macOS libs for the right arch and adding a Mac branch to each ThirdParty Build.cs.)
