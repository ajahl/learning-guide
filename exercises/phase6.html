Exercise 1 — Add a new config key to the lidar sensor

Starter~45 min Touches: GemstarLidarTest.ini → LidarSensorManager → SimpleLidar

Objective: Add a parameter the sensor honors — e.g. a MinRange that discards hits closer than N units (a real lidar has a dead zone).

Steps

Points nearer than MinRange vanish from the visualized cloud; raising the value carves a bigger hole around the sensor. No recompile needed to change the value — only to add the field.
Why this order? ini (data) → struct (carrier) → parse (read) → sensor (use). The same four hops apply to any new sensor parameter in GEMSTAR.
If you change only the ini value (not code) and it takes effect, but adding the field required a rebuild — why? (Values are data read at runtime; the struct field is compiled code.)

Exercise 3 — Add & log a field through the OSI conversion

Intermediate~1–1.5 hr Touches: ToOSI, the logger

Objective: Follow data into the OSI layer — the heart of GEMSTAR's interoperability — by populating one more field and confirming it in the log output.

Steps

Your value appears in the logged OSI message / trace. You've connected UE world state → OSI wire format — the exact bridge that makes GEMSTAR interoperable with other ASAM-OSI tools.
Heads up: this touches gemstar_interfaces, which depends on OSI/protobuf. On Windows those link and run, so you can do it live. On macOS they're Windows-only binaries in this repo, so do it as a code change + reasoned read unless you've ported the libs.
Why is ToOSI the single most strategic file to understand in GEMSTAR? (It's the one boundary every feature crosses to speak the common OSI language — change it and you affect logging, communication, and interop everywhere.)

Steps

A new feature plugin that the Root discovers, spawns, and configures on play — toggleable via UseExample. Replace the log with real work (implement a dispatcher/update interface) and it becomes a true producer or consumer like the built-in features.
Why this is the whole architecture in one exercise: manifest (loading) → Loader component (ignition) → Manager + IGEMSTARManager (the contract) → Use* flag + ini (config). Every GEMSTAR feature is this skeleton with real sensor/sim logic added.
Hub coupling, on purpose: step 5 edits the shared gemstar_interfaces — the one place features touch each other (the Use* / ClassName.Contains routing). It's the architecture's known trade-off; note it as you wire it.

Exercise 6 — Preload a feature's assets the Epic way (Asset Manager + Asset Bundles)

Advanced~2 hr Touches: a GameFeature plugin + GameFeatureData + DefaultGame.ini

Objective: Make a feature plugin declare the assets it needs and have them loaded before it activates — the idiomatic Unreal way (Asset Manager + Asset Bundles, triggered by a UGameFeatureAction on the plugin's GameFeatureData). Today GEMSTAR loads assets lazily — e.g. PointCloudVisualizer does LoadObject<UMaterial>(...) for its point-marker material, and VehicleManager::UpdateVehicleMap LoadObject<UClass>s each vehicle Blueprint under /ue_vehicle_feature/Blueprints/Vehicles on first spawn — which hitches and can miss assets. This is the standard fix.

How this relates to Phase 3. Phase 3 showed GEMSTAR spawns managers with a loader component, not a UGameFeatureAction. Asset preloading is the job UGameFeatureAction + the Asset Manager were actually designed for — so here you use the Epic idiom, and it coexists with the loader. "Bundle = what to load (declared on a data asset); Action = when to load it (the feature's activate/deactivate lifecycle)."

Follow Epic's guide

Steps

The feature's assets are resident before any manager runs, loaded on activation and unloaded on deactivation — no first-use hitch, no "asset not found" fallback. You've used the Asset Manager + Asset Bundles + a GameFeatureAction exactly as Epic's docs prescribe.
Why the bundle tag matters: without meta=(AssetBundles="Active") the soft pointers are just references the Asset Manager won't batch-load. The tag is what turns "a list of assets" into "a loadable set keyed to the feature's lifecycle."
Why preload via a GameFeatureAction instead of just LoadObject in BeginPlay? (Activation-time loading is declarative, pins the assets for the feature's whole active lifetime, unloads cleanly on deactivate, and is cook-aware — the lazy load is none of those.)
