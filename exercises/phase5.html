Exercise 1 — Anatomy of an OSI message (read protobuf-generated headers)

Intermediate~40 min Skills: protobuf, OSI message structure

Objective: Learn to read a .pb.h file and recognize the OSI message hierarchy GEMSTAR serializes to.

Steps

What to look for Classes like Vector3d, Orientation3d, Dimension3d, Spherical3d, Timestamp. Each is a protobuf message: note the generated accessors — x() / set_x() for scalars, mutable_position() for nested messages. This getter/setter pattern is how all OSI fields are read and written.
What to look for GroundTruth = the complete, perfect world state (all moving/stationary objects, lanes, lights). SensorView wraps a GroundTruth (often filtered to a sensor's viewpoint) plus the sensor's mounting/config. A repeated field (moving_object() list) is protobuf's array — note _size() and add_*() accessors.
Sketch the nesting: SensorView → GroundTruth → MovingObject[] → BaseMoving → {Vector3d position, Dimension3d, Orientation3d}. If you can draw that, you can read any OSI message.

Exercise 2 — Trace the UE → OSI bridge in ToOSI

Intermediate~45 min Skills: coordinate mapping, the integration boundary

Objective: See exactly how a UE actor's transform becomes an OSI message — the single most strategic file in GEMSTAR.

Steps

What to look for — and a subtlety It reads the actor's location/rotation and writes them into an OSI BaseMoving via the protobuf setters you met in Exercise 1. Subtlety to hunt for: any unit or axis conversion — UE uses centimeters and a left-handed Z-up frame; OSI uses metres and a right-handed frame. Look for a /100, a sign flip, or a yaw adjustment. This mismatch is a classic integration bug source.
Why is ToOSI the boundary every feature crosses? (It's the one place UE world state becomes the common OSI language — logging, communication, and interop all funnel through it.)

Exercise 3 — Find where OSI traces are written

Intermediate~30 min Skills: serialization, trace files, the logger

Objective: Connect the message types to the files GEMSTAR actually produces in Output/ — exactly what the “ASAM OSI trace file” video describes.

Steps

What to look for OSI traces are typically length-prefixed binary protobuf: each record = a 4-byte size then the serialized message bytes. That's why a player needs to know the message type to read them back. Confirm the type the logger writes, and you've closed the loop: actor → ToOSI → message → serialized bytes → file in Output/.
Explain to an imaginary colleague what a .osi trace contains and how another ASAM-OSI tool could replay it. If you can, you understand GEMSTAR's interoperability story.
