↪ The assignmentBefore the shuttle can detect a pedestrian, you have to be able to place an object in a world and press play. This chapter is exactly that — learning to see what the engine sees.
One-time setup: Launch Unreal Editor — Windows: from the Epic Launcher, or Engine\Binaries\Win64\UnrealEditor.exe of a source build; macOS: …/Engine/Binaries/Mac/UnrealEditor.app. In the Project Browser → Games → Blank, Blueprint (not C++), Desktop, no starter content needed. Name it UEScratch. This is your sandbox for Phases 1–3.
Two words you'll keep meeting: an Actor is a thing placed in the level (a vehicle, a sensor, the GEMSTAR Root). A Component attaches to an Actor to add behavior or data (a mesh, a camera — or GEMSTAR's GEMSTAR_*_Loader, which is a component on the Root). Actor = the object; Component = a part bolted onto it. You'll build both; the loader pattern in Phase 3 is component-based.

Exercise 1 — A cube that rotates

Beginner~20 min Skills: Actors, Blueprints, Tick, viewport

Objective: Place an object in the level and drive it with a Blueprint that runs every frame.

Steps

The cube spins continuously around its vertical axis. Stop play and it returns to rest — Tick only runs during play.
Why Yaw = 2.0 and not a "speed"? You're adding 2° per frame, so the speed depends on framerate. The proper fix — multiply by Delta Seconds (the Tick node's output) — is Exercise 2 of Phase 2 in spirit. Note that thought now.
Troubleshooting
Cube invisible after dragging inYou added the component to the Blueprint but its mesh slot may be empty — select the Cube component, in Details set Static Mesh = Cube.
Nothing rotatesDid you wire Event Tick (not BeginPlay)? Did you Compile after editing?
Can't find AddActorLocalRotationUncheck "Context Sensitive" in the node search, or drag from the Tick exec pin specifically.
Explain out loud: what is the difference between the StaticMeshActor you first placed and your BP_Spinner Blueprint class? (Answer: BP_Spinner is a reusable class/blueprint with behavior; the StaticMeshActor was a bare instance.)

Exercise 2 — Logging on BeginPlay and every Tick

Beginner~15 min Skills: lifecycle events, Output Log, Print String

Objective: Feel the difference between BeginPlay (once) and Tick (every frame) — the exact distinction GEMSTAR's managers rely on.

Steps

BeginPlay ran once appears a single time at the start. The Delta Seconds value prints continuously (a small number like 0.016 ≈ 1/60s) — one line per frame.
Connect to GEMSTAR: ASUMOManager and AEsminiManager step their simulators inside Tick, and do one-time setup in BeginPlay. You just reproduced that pattern in miniature.
Why does the Delta Seconds number get bigger if you drag the editor window or stress your machine? (Because each frame takes longer, so more real time elapses per Tick.)

Exercise 3 — A DataTable and reading a row

Beginner+~25 min Skills: Structs, DataTables, CSV import, lookups

Objective: Build a DataTable and look up a row — the exact mechanism ue_vehicle_feature uses to map a vehicle type to a spawnable Blueprint.

Steps

On play, the Output Log / screen prints the DisplayName you entered for the Car row.
Bonus: Change Row Name to a value that doesn't exist (e.g. Truck) and observe the Row Found pin go false. GEMSTAR must handle exactly this — a simulator sending an unknown vehicle type.
In GEMSTAR's vehicle feature, what would the equivalent row contain instead of TopSpeed? (A reference to a vehicle Blueprint/class to spawn — open VehicleTableRow.h later to confirm.)
