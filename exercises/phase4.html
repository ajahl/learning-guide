Phase 4 — Guided Code Read: Worksheet

A structured trace through GEMSTAR's lidar → visualizer path. For each file, predict what data enters and leaves, then open it and check. On Windows you can also run it live; on macOS it works as a paper trace — either way, no build is required to do the reading.

↪ The assignmentThe heart of the assignment. Trace this ray and you're watching the exact mechanism that decides whether the shuttle sees the pedestrian — or misses them.
How to work this: read each "Predict" prompt and answer it in your head (or notes) before opening the file. Then open it, confirm, and reveal the "What to look for" to check yourself. The goal is to hold the whole data path in your head by the end.

Station 1 — The inputs: GemstarLidarTest.ini

What to look for A [lidar:sensorN] section per sensor: MountingPosition X/Y/Z & Orientation (where the sensor sits), NumberOfRaysHorizontal/Vertical (density), FoVHorizontal/Vertical (coverage), Range & DistanceHalfFalloff. Rays = density; FOV + Range = shape/extent.

Station 2 — The orchestrator: AGEMSTARRoot

What to look for Config reads (use-flags like UseLidar), and the RegisterDataflow / MountOn calls from the IGEMSTARManager interface (Phase 2!). The Root caches managers and connects producers to consumers. It does not contain lidar math — it wires, it doesn't compute.

Station 3 — Spawn & configure: ALidarSensorManager

What to look for An FSensorConfigData-style struct populated from the ini, and a spawn loop creating one SimpleLidar per configured sensor, applying mounting transform + parameters. Enters: config. Leaves: spawned, configured sensor actors.

Station 4 — The sensor: SimpleLidar

What to look for Nested loops over horizontal × vertical rays computing directions from FOV, then a line trace (LineTraceSingleByChannel or similar) per ray; hits become points. Enters: config + world geometry. Leaves: a set of 3D points (the cloud).

Station 5 — The hand-off: dispatcher + update interface

What to look for A dispatcher holding a list of receivers and a method to broadcast; an update interface the receivers implement. The producer pushes by calling the update method on each registered receiver. This is your Phase 2 Execute_ pattern at work — the sensor never knows it's talking to the visualizer.

Station 6 — The sink: AVisualizerManager → PointCloudVisualizer

What to look for It implements ISensorDataUpdate (registered as a receiver), converts OSI points to world positions, and renders markers. Enters: sensor data via the update callback. Leaves: visible debug points.

Bonus A — Shortcut in: the *Tester classes

Each feature ships a tester/mockup that implements the dispatcher or update interfaces directly — the easiest way to see one side of the pub-sub in isolation, without the whole pipeline.

Why are these the fastest way to understand an interface? (They show exactly which methods a receiver must implement — the contract made concrete, with nothing else in the way.)

Bonus B — Trace the camera & vehicle paths yourself

Same producer→consumer shape as lidar→visualizer, different mechanics. Apply the exact predict-then-check method.

Vehicle is the mirror image of lidar: lidar produces sensor data; vehicle consumes simulator data. Both use the same dispatcher/update interface machinery. If you see that symmetry, you've got the architecture.

Final — Reconstruct & (optionally) tweak

You pass this phase when you can name every box from memory and say what enters/leaves each. That single mental model unlocks all the other features — they're the same shape with different producers/consumers.
