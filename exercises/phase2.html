Phase 2 — C++ in Unreal: Exercises

Use a C++ scratch project. These build the exact constructs GEMSTAR uses — UPROPERTY, the Actor lifecycle, and UInterfaces (the pattern the whole codebase is wired with).

Steps done
0 / 0
↪ The assignmentThe shuttle's sensors and the traffic simulator never call each other directly — they meet through interfaces. Master that here and the rest of GEMSTAR stops looking like magic.
One-time setup: New project → Games → Blank, but this time choose C++. Name it CppScratch. Replace CppScratch with your real project name (the API macro CPPSCRATCH_API follows the project name in UPPER_CASE). After adding C++ classes, build from Xcode (or let the editor's "compile" button rebuild).
Platform note: these C++ exercises compile on any OS (Windows/macOS/Linux) because they use only Core/Engine — no third-party libs. GEMSTAR's real plugins are different: they build on Windows but need a Mac/Linux port (see the platform table in the main guide, and Phase 2.5).

Exercise 1 — An Actor with an editable UPROPERTY

Beginner~30 min Skills: UCLASS, UPROPERTY, Tick, reflection → editor

Objective: Expose a C++ variable to the editor and use it in Tick — see the reflection system bridge C++ and the editor UI.

Steps

// SpinningActor.h
UCLASS()
class CPPSCRATCH_API ASpinningActor : public AActor
{
    GENERATED_BODY()
public:
    ASpinningActor();

    // EditAnywhere → shows up in the Details panel; the value is reflected.
    UPROPERTY(EditAnywhere, Category = "Spin")
    float SpinRateDegPerSec = 90.f;

protected:
    virtual void Tick(float DeltaTime) override;
};
// SpinningActor.cpp
#include "SpinningActor.h"

ASpinningActor::ASpinningActor()
{
    PrimaryActorTick.bCanEverTick = true;   // opt in to Tick
}

void ASpinningActor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);
    AddActorLocalRotation(FRotator(0.f, SpinRateDegPerSec * DeltaTime, 0.f));
}
The SpinRateDegPerSec field appears in the Details panel (that's UPROPERTY(EditAnywhere) working). At 360 it does one full turn per second, regardless of framerate.
Troubleshooting
Property not in Details panelYou likely forgot UPROPERTY(EditAnywhere), or didn't rebuild. A plain C++ member without the macro is invisible to the editor.
Build error: redefinition / API macroCPPSCRATCH_API must match your project's module name in caps. Check another generated header.
Doesn't tickPrimaryActorTick.bCanEverTick = true; must be in the constructor.
Why does the editor even know SpinRateDegPerSec exists? (The UPROPERTY macro feeds Unreal's reflection system, which the Details panel reads — plain C++ members are invisible to it.)

Exercise 2 — A UInterface, implemented and called ★ core GEMSTAR pattern

Intermediate~45 min Skills: UINTERFACE, BlueprintNativeEvent, Execute_, Implements<>

Objective: Define a contract, implement it on an Actor, and call it through the interface without knowing the concrete type. This is precisely how IGEMSTARManager / IVehicleDataDispatcher decouple GEMSTAR's plugins.

Steps

// Pingable.h
UINTERFACE(MinimalAPI, Blueprintable)
class UPingable : public UInterface
{
    GENERATED_BODY()
};

class IPingable
{
    GENERATED_BODY()
public:
    // BlueprintNativeEvent: has a C++ default AND can be overridden in Blueprint.
    UFUNCTION(BlueprintNativeEvent, BlueprintCallable, Category = "Ping")
    void Ping(const FString& From);
};
Why two classes? UPingable is the reflection stub Unreal needs; IPingable is the one your classes actually inherit. Every UE interface — including GEMSTAR's — comes in this U.../I... pair.
// PingResponder.h
#include "Pingable.h"
UCLASS()
class CPPSCRATCH_API APingResponder : public AActor, public IPingable
{
    GENERATED_BODY()
public:
    // For a BlueprintNativeEvent named Ping, you override Ping_Implementation:
    virtual void Ping_Implementation(const FString& From) override;
};
// PingResponder.cpp
#include "PingResponder.h"
void APingResponder::Ping_Implementation(const FString& From)
{
    UE_LOG(LogTemp, Warning, TEXT("Ping received from %s"), *From);
}
// Calling code — note we hold an AActor*, not an APingResponder*
if (TargetActor && TargetActor->Implements<UPingable>())
{
    // Execute_Ping routes to C++ override OR a Blueprint override, correctly.
    IPingable::Execute_Ping(TargetActor, TEXT("Pinger"));
}
Ping received from Pinger logs as a warning. The caller never referenced APingResponder — only the UPingable contract.
Troubleshooting
Linker error on PingFor BlueprintNativeEvent you must implement Ping_Implementation, not Ping.
Implements<> always falseThe actor must inherit public IPingable AND be a UObject-derived class. Rebuild after adding the base.
Why not just call Ping() directly?Direct calls bypass Blueprint overrides and crash if the object doesn't implement the interface. Always use Execute_.
Open GEMSTARManager.h in GEMSTAR and find the UINTERFACE/I... pair and a ...Implementation method. You should now recognize every piece. This is the single most important transfer in the whole plan.

Exercise 3 — Two modules, one depending on the other

Intermediate~40 min Skills: modules, Build.cs, public/private deps

Objective: Split code into two modules and link them via .Build.cs — the same dependency mechanism that lets every GEMSTAR feature depend on GEMSTAR_Interfaces.

Steps

// CppScratch.Build.cs
PublicDependencyModuleNames.AddRange(new string[] {
    "Core", "CoreUObject", "Engine", "InputCore",
    "CoreContracts"          // <-- now PingResponder can #include "Pingable.h"
});
The project builds with the interface defined in one module and used in another. Removing "CoreContracts" from the deps produces an "unresolved include / unknown type" error — proving the link is real.
Public vs Private deps: use PublicDependencyModuleNames when your headers expose the other module's types (as here — APingResponder.h includes Pingable.h). Use PrivateDependencyModuleNames when only your .cpp needs it.
Why does GEMSTAR put all interfaces in one GEMSTAR_Interfaces module that every feature lists as a public dependency? (So features share contracts without depending on each other — the hub-and-spoke you saw in Phase 0.)
